## 逐项详情

### 1. 导入脚本解析文件名崩溃

**症状**：`Error loading english_grammar.txt: invalid literal for int() with base 10: 'english'`

**根因**：`import_edu_data.py` 从医疗版复制，按 `{数字前缀}_{类型}_{疾病}.txt` 格式解析文件名，`int('english')` 失败。

**修复**：重写 `import_edu_data.py` 的 `parse_filename()` 函数，支持 `{subject}_{topic}.txt` 格式。

**文件**：`knowledge/scripts/import_edu_data.py`

**修复**：`pip install 'pymilvus[milvus_lite]'`

### 3. events 文件名带日期后缀

**症状**：`ModuleNotFoundError: No module named 'swarm.events'`

**根因**：原始文件名为 `events_20260428_231035.py`，`shared_context.py` import `.events`

**修复**：`mv events_20260428_231035.py events.py`

### 4. Agent 抽象方法未实现

**症状**：`Can't instantiate abstract class TutorAgent without an implementation for abstract method 'register_tools'`

**根因**：`BaseAgent.__init__` 要求 `config`（必传参数），且 `register_tools` 是 abstract method，三个教育 Agent 都没实现。

**修复**：三个 Agent 各补了 `config={"temperature": 0.7, "max_iterations": 10}` 和空的 `register_tools`。

**文件**：`agents/tutor_agent.py`, `agents/progress_agent.py`, `agents/assess_agent.py`

### 5. Skill 注册函数为 None

**症状**：`Skill execution failed: assess_level - 'NoneType' object is not callable`

**根因**：`register_tools` 里 `function=None`，Skill 实际函数没注入。

**修复**：Agent 改为继承 `SkillRegistryMixin`，`register_tools` 改成 `self.register_all_skills()`——自动从 `.claude/skills/` 扫描加载。

**文件**：三个 Agent 文件 + 创建 9 个 `SKILL.md`（discover_skills 需要）

### 6. reasoning_content 未回传（豆包/DeepSeek thinking 模式）

**症状**：`Error code: 400 - 'The reasoning_content in the thinking mode must be passed back to the API.'`

**根因**：豆包 seed 和 DeepSeek R1 在 thinking 模式下返回 `reasoning_content`，必须在下一轮 assistant 消息中原样传回。现有代码完全无视这个字段。

**修复**：
- `LLMResponse` 新增 `reasoning_content` 字段
- `chat_with_tools` 从 `message.reasoning_content` 捕获
- `agent_loop._create_assistant_message_with_tools` 回传该字段

**文件**：`core/llm_client.py`, `core/agent_loop.py`

---

### 7. Milvus Lite gRPC 日志刷屏

**症状**：`too_many_pings` `ENHANCE_YOUR_CALM` GOAWAY 日志

**根因**：Milvus Lite 的 gRPC keepalive ping 默认 10s 间隔太频繁，被服务端限流。不影响功能，但日志污染严重。

**修复**：在 `milvus_kb.py` 顶部 `os.environ.setdefault('GRPC_VERBOSITY', 'ERROR')` 抑制 gRPC 内部日志。

**文件**：`knowledge/milvus_kb.py`

---

### 8. Surrogate 字符导致全链路 UTF-8 编码崩溃

**症状**：`'utf-8' codec can't encode characters: surrogates not allowed`

**根因**：emoji 或特殊 Unicode 字符在某些边界被截断，产生 lone surrogate 码点（U+D800-U+DFFF）。`json.dumps(ensure_ascii=False)` 将其直接写入字符串，后续 `str.encode('utf-8')` 崩溃。

**修复**：
- 在 `llm_client.py` 新增 `_sanitize()` 递归清理函数，所有进出 LLM 的消息串都过一遍
- `short_term.py` 的 `add_message` 入口消毒
- `long_term.py` 的 Mem0 进出消毒
- `agent_loop.py` 的 `json.dumps` 加 `errors='replace'`

**文件**：`core/llm_client.py`, `core/agent_loop.py`, `memory/short_term.py`, `memory/long_term.py`

---

### 9. Mem0 方法名不匹配

**症状**：`'LongTermMemory' object has no attribute 'search'`

**根因**：`StudentProfileManager` 调用 `self.long_term_memory.search()` / `.add()`，实际方法名为 `search_similar_sessions()` / `add_session_summary()`。

**修复**：改为通过 `self.long_term_memory.mem0`（底层 Mem0 MemoryClient）直接调用 `search` 和 `add`。

**文件**：`memory/student_profile.py`

---

### 10. 多导师模式无法触发

**症状**：问"帮我全面评估物理力学的知识体系"仍然走单导师

**根因**：LLM（豆包）分解任务只返回 1 个 subtask，`len(subtasks) >= 2` 条件不满足。

**修复**：在 `swarm_coordinator.py` 加关键词规则兜底——检测到"全面评估""知识体系""查漏补缺"等关键词时，跳过 LLM 判断直接拆成 2-3 个 Agent。

**文件**：`swarm/swarm_coordinator.py`

---

### 11. Swarm 超时 258 秒

**症状**：多导师模式下总耗时 258 秒，assess_agent 单 Agent 就 140 秒

**根因**：三个问题叠加——
1. `max_tool_calls=2` 太低，LLM 一次请求就要 3-4 个工具
2. 达到上限后死循环：LLM 说要调工具 → 代码说"已达上限" → LLM 继续要工具 → 重复 10 轮
3. 每轮都在刷 "已达到最大 Skill 调用次数限制" 日志

**修复**：
- `max_tool_calls`: 2 → 5
- 达到上限后最多强制 2 轮，还不出 `finish_reason="stop"` 直接 break
- Swarm 超时：90s → 180s

**文件**：`core/agent_loop.py`, `swarm/swarm_coordinator.py`

---

### 12. 跳过 tool 后 API 报 insufficient tool messages

**症状**：`Error code: 400 - insufficient tool messages following tool_calls message`

**根因**：修复 #11 时，超标 tool 直接 `continue` 跳过，但没给 `tool_call_id` 加结果消息。OpenAI API 要求每个 `tool_call_id` 必须有一条 tool message 对应。

**修复**：跳过执行时补一条假结果消息：`{"skipped": True, "reason": "已达最大调用次数限制"}`

**文件**：`core/agent_loop.py`
